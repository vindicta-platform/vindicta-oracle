## Clarifications

### Session 2026-02-23

- Q: Fallback Cache lifecycle — when is it populated, how long does it stay valid, what triggers invalidation? → A: Write-through cache. Populated opportunistically on every successful RAG retrieval. No hard TTL — always as fresh as the last successful call.
- Q: Who is responsible for parsing raw markdown rule segments into structured `UnitStats` objects (toughness, wounds, points, weapons)? The RAG server returns markdown chunks, but the List Grader needs numeric fields. → A: The oracle's `GradingDataAssembler` is responsible for parsing. The foundation returns raw markdown; the oracle applies regex/heuristic extraction to produce `UnitStats`. This keeps foundation generic and oracle domain-aware.
- Q: What is the default context window budget for `RulesContextPackage` (FR-008)? How is it measured — tokens, characters, or bytes? → A: Budget is measured in **characters** for simplicity in the MVP. Default budget is 32,000 characters (~8K tokens). Configurable per-agent via `ContextAssembler` constructor.
- Q: Where does the `[RULE-ID]` citation format come from — is it embedded in the RAG segments or generated by agents? → A: The `segment_id` (UUID) from the foundation's `RulesSegment` is the canonical rule identifier. Agents embed `[RULE-{segment_id}]` citations. Rule-Sage verifies these against the `RulesContextPackage.segments` list by matching the UUID portion.
- Q: Does `batch_query` (FR-004) issue a single MCP tool call with multiple unit names, or multiple sequential calls? → A: Single MCP tool call. The foundation's `search_40k_rules` tool must accept a `unit_names: list[str]` parameter. If the foundation only supports single-unit queries in the MVP, the oracle's `RAGClient.batch_query()` will internally loop and aggregate, but the interface remains batch-oriented.
- Q: How does the system resolve edge case EC-005 (two versions of the same datasheet with identical timestamps)? → A: Tiebreak by `segment_id` lexicographic order (deterministic). Log a warning indicating the ambiguity. This is a data quality issue in the ingest pipeline, not an oracle concern.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Grounded Debate with RAG-Backed Rules (Priority: P1)

During a Council Protocol debate session, agents need access to exact weapon profiles, unit abilities, and current errata so that arguments reference actual game rules rather than stale or hallucinated data. The Debate Engine queries the RAG rules server on behalf of each agent before (and during) a debate round, injecting retrieved rule segments directly into the agent's context. This ensures every citation (`[RULE-ID]`) can be verified against a live, versioned source of truth.

**Why this priority**: Groundedness is the single most important quality signal. Without live rule retrieval, the Rule-Sage cannot distinguish a legitimate citation from a hallucination, and the entire council verdict loses trustworthiness.

**Independent Test**: Can be fully tested by initiating a debate round for a known unit and verifying that the debate transcript contains citations whose `[RULE-ID]` tags match entries returned by the RAG server for that unit.

**Acceptance Scenarios**:

1. **Given** a debate round involving a unit with a recently errata'd weapon profile, **When** the Debate Engine requests rules from the RAG server, **Then** the retrieved segments reflect the latest errata version and the agent's argument references the corrected profile.
2. **Given** the RAG server is unreachable, **When** the Debate Engine attempts to fetch rules, **Then** the engine falls back to a cached rules snapshot and logs a warning so the debate can still proceed.
3. **Given** an agent cites a rule that is not present in the retrieved segments, **When** Rule-Sage audits the transcript, **Then** the citation is flagged as ungrounded and the agent's round is penalised in scoring.

---

### User Story 2 - Dynamic List Grading Against the Current Meta (Priority: P1)

The Army List Grader fetches unit statistics and point costs from the RAG server to evaluate a submitted list against the current competitive meta. Rather than relying on a static snapshot, each grading request pulls the latest unit data so that point increases, datasheet rewrites, and balance changes are automatically reflected in the grade.

**Why this priority**: A grade based on outdated points or stats is worse than no grade at all — it actively misleads players. Dynamic retrieval is co-equal with debate groundedness.

**Independent Test**: Can be fully tested by submitting a list containing a unit whose point cost recently changed and verifying that the grade calculation uses the updated cost.

**Acceptance Scenarios**:

1. **Given** a list submission containing units from a faction, **When** the List Grader evaluates the list, **Then** the grading pipeline retrieves each unit's current stats and point cost from the RAG server and uses those values for the score calculation.
2. **Given** the RAG server returns multiple versions of a unit's datasheet, **When** the List Grader processes the results, **Then** only the most recent version is used for grading.
3. **Given** a unit in the submitted list cannot be found in the RAG server, **When** the List Grader encounters the unknown unit, **Then** the system returns a partial grade with a clear warning identifying the unresolved unit(s).

---

### User Story 3 - Pre-Debate Context Assembly (Priority: P2)

Before a debate session begins, the Debate Engine pre-fetches all rules relevant to every unit in the candidate list and assembles them into a structured context package. This pre-assembled context is distributed to each agent at round start, reducing per-turn latency and ensuring all agents argue from the same factual base.

**Why this priority**: Bulk pre-fetch reduces total RAG calls and guarantees consistency across agents during a single debate. However, the system can still function with per-turn retrieval (P1), making this an optimisation layer.

**Independent Test**: Can be tested by measuring RAG server call count and debate latency for a 3-round debate with and without pre-fetch enabled, and verifying the pre-fetch path issues fewer total requests.

**Acceptance Scenarios**:

1. **Given** a candidate list with 10 unique units, **When** the Debate Engine enters the pre-fetch phase, **Then** the system issues a batch query for all 10 units and receives consolidated results in a single round-trip.
2. **Given** the pre-fetched context exceeds the agent's context window limit, **When** the context is distributed, **Then** the system applies relevance-ranked truncation so the most pertinent rules are included within the window budget.

---

### User Story 4 - Meta Snapshot for Comparative Grading (Priority: P3)

The List Grader retrieves aggregate meta statistics (e.g., faction win rates, popular unit choices) from the RAG server to position the submitted list within the broader competitive landscape. This enables qualitative commentary like "This list lacks anti-tank in a meta dominated by vehicle-heavy factions."

**Why this priority**: Meta context enriches the grade with competitive insight but is not required for a mechanically accurate score. The system can grade purely on stats without this layer.

**Independent Test**: Can be tested by grading the same list with and without meta context enabled, and verifying the meta-aware grade includes contextual commentary referencing current faction trends.

**Acceptance Scenarios**:

1. **Given** a submitted list for a faction, **When** the List Grader fetches meta context from the RAG server, **Then** the grading response includes a `meta_context` section referencing current competitive trends for that faction.

### Edge Cases

- What happens when the RAG server returns stale data because the ingest pipeline has not yet processed a recent FAQ or errata update?
- How does the system behave when the RAG server returns zero results for a highly specific or misspelled unit query?
- What happens when a batch pre-fetch for a large list (20+ unique units) exceeds the RAG server's response size limit?
- How does the Debate Engine handle a mid-debate RAG server outage after the pre-fetch phase has already completed?
- What happens when two versions of the same unit datasheet are returned with identical timestamps? → **Resolved**: Tiebreak by `segment_id` lexicographic order (deterministic); log a warning. This is a data quality issue in the ingest pipeline.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The Debate Engine MUST query the RAG rules server to retrieve weapon profiles, unit abilities, and errata before distributing context to debate agents.
- **FR-002**: The List Grader MUST query the RAG rules server to retrieve current unit statistics and point costs for every unit in a submitted army list.
- **FR-003**: When multiple versions of a rule segment are returned, the system MUST use only the most recent version (determined by insertion timestamp). When timestamps are identical, the system MUST tiebreak by `segment_id` lexicographic order and log a warning indicating the ambiguity.
- **FR-004**: The system MUST support a batch query mode that retrieves rules for multiple units via a single MCP tool call (accepting `unit_names: list[str]`). If the foundation server only supports single-unit queries in the MVP, the oracle's client MUST internally loop and aggregate while preserving the batch-oriented interface.
- **FR-005**: When the RAG server is unreachable, the system MUST fall back to a locally cached rules snapshot and log a degraded-mode warning on the debate transcript or grading response. The cache is populated via write-through on every successful RAG retrieval — no hard TTL, always as fresh as the last successful call.
- **FR-006**: The Debate Engine MUST inject retrieved rule segments into the agent context so that Rule-Sage can verify `[RULE-{segment_id}]` citations against the `RulesContextPackage.segments` list by matching the UUID portion of each citation tag.
- **FR-007**: The List Grader MUST return a partial grade with explicit warnings when one or more units cannot be resolved against the RAG server.
- **FR-008**: The system MUST apply relevance-ranked truncation when retrieved context exceeds the agent's context window budget. The budget is measured in characters with a default of 32,000 characters (~8K tokens), configurable per-agent.
- **FR-009**: The system MUST record which RAG segments were used in each debate round or grading session for auditability and reproducibility.
- **FR-010**: The oracle's `GradingDataAssembler` MUST parse raw markdown rule segments returned by the foundation's RAG server into structured `UnitStats` objects (toughness, wounds, points, weapons) using regex/heuristic extraction. The foundation remains generic; domain-aware parsing is an oracle responsibility.

### Key Entities

- **RAG Query**: A structured search request containing unit names, weapon keywords, or rule topics. Includes a query type (unit stats, weapon profile, errata, meta context) and an optional version filter.
- **Rules Context Package**: An assembled set of rule segments distributed to debate agents for a session. Contains the source RAG segment IDs (UUIDs matching foundation `RulesSegment.segment_id`), retrieval timestamps, and relevance scores for each included segment. Subject to a configurable character budget (default: 32,000 characters).
- **UnitStats**: A structured object (toughness, wounds, points, weapons) produced by the oracle's `GradingDataAssembler` from raw markdown rule segments. The canonical parsed representation used for numeric grading calculations.
- **Grading Data Bundle**: The subset of RAG-retrieved data (unit stats, point costs, meta statistics) used by the List Grader for a single evaluation. Includes provenance metadata linking each value back to its RAG segment ID.
- **Fallback Cache**: A write-through cache populated on every successful RAG retrieval, keyed by unit or rule topic. No hard TTL — freshness matches the last successful call. Used automatically when the RAG server is unavailable.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of debate transcripts contain only `[RULE-ID]` citations that can be traced back to a RAG-retrieved segment — zero ungrounded citations in a passing audit.
- **SC-002**: List grades computed with live RAG data reflect point cost changes within one ingest cycle of the change being published on the source website.
- **SC-003**: Pre-debate context assembly for a typical list (10-15 unique units) completes in under 3 seconds.
- **SC-004**: When the RAG server is unavailable, the system still produces a debate transcript or list grade using cached data, with a visible degraded-mode indicator, in 100% of fallback scenarios.
- **SC-005**: 95% of unit lookups by the List Grader resolve successfully against the RAG server for supported factions.
